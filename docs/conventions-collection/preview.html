Structural Resilience & Engineering Discipline

10
A1 Dependency seam — every external dep behind one config-swappable interface Needs authoring
Every call to an external service goes through a single stable interface chosen by config. Swapping providers is a config change, never a call-site change. Enforced by construction (grep-the-tree test forbids SDK calls outside the seam). Convergent: independently derived in smart_wardrobe and job-curator without either naming it.
Jean deep
Upfront indirection + adapter/factory layer for deps you may never swap. smart_wardrobe built it 7 sprints before the first alternate provider.
A2 Soft-degrade by default — failure degrades, never crashes Needs authoring
When an external dep fails or is absent, the feature falls back to a deterministic/manual path and stays usable. Failures logged, not surfaced as errors. Convergent: both smart_wardrobe and job-curator.
Jean deep
Load-bearing: resilience can mask whether the premium path ever works. smart_wardrobe stayed usable on a dead LLM key for 4 sprints while AI was never verified live. Must pair with a live-verification harness (C1) and a visible "degraded" signal.
B1 Fail-fast config guards — invariants as hard boot errors Needs authoring
A dangerous misconfiguration refuses to boot, names the flag, states the rule. The invariant is enforced in code, not left as a convention to remember.
Jean deep
The guard is a moving target — every posture change means re-scoping it and updating tests in lockstep.
B2 Deletion completeness — real delete purges all stores + external identity Needs authoring
"Delete" purges all stores (DB + object storage + auth identity), ordered so mid-failure is retryable, fails loud (502) rather than claiming success on partial. Every new table added to purge paths. Extends 02-data-protection.md.
Jean deep
Deletion can now hard-fail when infra creds missing; requires live smoke tests.
B3 LLM provider choice is a data-protection decision Needs authoring
Local/self-hosted inference eliminates the data-egress + training question entirely. The privacy story becomes provider-conditional. Extends 02-data-protection.md.
Jean deep
Local-first trades privacy simplicity for compute/quality limits.
J5 Structural-invariant-over-promise Needs authoring
Enforce a security/integrity property by construction (a surface that never holds a value can't leak it; tooling with no write path can't post). Audit recovery/rerun paths against the same invariant.
Jean workspace
J13 Enforce-at-authoring-time — guards over sweeps Needs authoring
A convention worth having is a self-executing write-time guard (PreToolUse), not a recurring cleanup sweep. "A convention enforced after the fact generates cleanup work forever."
Jean workspace
Upfront guard-building + false-positive risk. Mitigate by matching structural shape.
J14 Multi-writer JSON-store hardening contract Needs authoring
flock (EX write / SH read) + atomic temp-file + os.replace + strict-load-on-write (corrupt aborts, never wipes) + tolerant-load-on-read. Six ~/.fleet modules independently re-derived the same discipline.
Jean workspace
Author as both a rule AND a shared helper — prose-only promotion invites copy-paste drift.
J15 Hardcode-then-parameterize Needs authoring
Spike with literals to verify a path, then lift every dynamic value to config before it counts as done.
Jean workspace
J16 Local-LLM structured-JSON output contract Needs authoring
Never parse free-text-with-markers from a small local model: call with format: json, make needed fields required, validate leniently, sanitize extracted values after parse, fail soft.

Verification & Honesty

8
C1 Mocked tests are green-but-meaningless for the live path Sharpen
Tests mock external providers → green proves wiring/fallback, never live integration. Each integration needs a named operator smoke recipe AND a live-verification harness that exits non-zero on degrade. Strongest convergent signal — near-identical wording in both repos.
Jean deepJean workspace
A permanently-manual step CI can't gate; the harness is built so it's harder to claim success.
C2 Mutation-prove a lock/regression test actually "bites" Sharpen
A test meant to lock a behavior isn't trusted until you mutate the code to break it and confirm the test goes red, then revert. Hazard: don't run mutation via git checkout <file> — that clobbered uncommitted edits in Sprint 7.
Jean deep
C3 Evidence over assertion — including the agent's own claims Sharpen
Before planning around a status/failure claim, probe the primary source. Apply the same skepticism to the agent's own statements. Most-cited principle in the entire record.
Jean deepJean workspace
Probing costs a step before acting on any claim; on quota-bound keys the probe spends the scarce resource.
J1 Watched-live bar — green tests are not the greenlight Needs authoring
A human-watched live run (zero freeze, real behavior observed) is the ship gate for any interactive surface. Single most-cited convention in Jean's record (174/268 tests green while 2 camera-breaking live bugs shipped, three separate times).
Jean workspace
Someone must actually watch a live run before every ship — slower than trusting CI.
J2 Adversarial pre-ship second-set-of-eyes Needs authoring
Anything shippable gets an independent adversarial pass first (different model or non-author), returning questions-to-verify, never verdicts — evidence decides. skeptic.py enforces questions-only in code.
Jean workspace
A warm second model (~25s/call) + discipline to run it by default.
J4 Honesty labels — ship the real, label the rest Partial
Real surfaces ship; prototype/snapshot surfaces carry a provenance chip. "Danger" tone reserved for genuinely broken. Currently 04-docs.md covers overclaim-scrub for Public tier only — needs extension.
Jean workspace
J16b Local-vs-hosted model selection rule Needs authoring
Local small model for cheap, high-frequency facilitation work; hosted model for real reasoning; and anything deterministic (roster lookup, owner matching) must actually be deterministic, not model-generated.
Jean workspace
J16c A review loop cannot audit taste Needs authoring
Correctness questions route to an auditor; editorial and taste calls stay with the owner. A loop will never converge on taste.

Operator-Side Conventions — Jean

Setting the mode

  • "do it for me" — flips from advisor to runtime; stops counseling, starts executing.
  • "don't act, tell me" / "just tell me" — holds all execution; answer only.
  • "review this" vs "help me build this" — outside-critic vs collaborator.
  • "did we actually…" / "verify" — check live state, not memory.

Authorizing an action

  • "go" / "yes" / "submit" / "proceed" — per-action authorization. Silence ≠ yes.
  • "in advance the answer is yes" — standing authorization.

Delegating

  • "make me a prompt for FM" — paste-ready routing block.
  • Batch it — approve multiple in one message.
  • Name the agent — "have GC look it over."

Being interviewed

  • "ask me one question at a time" — wall-of-questions → real interview.
  • Quick pick + a note; "Other" is always there.

Keeping it honest

  • Push back when it sounds too clean — "why are you celebrating."
  • "lets not get ahead of ourselves" — parks projecting.
  • Correct scope in my own words — becomes a durable rule.

Commands & rituals

  • /roundtable for team meetings → owned work; /broadcast for all-agents; /fleet and /routine for the board; /compact with keep-list.

Through-line: short and explicit beats long and hedged. Best inputs are three words.

Operator-Side Conventions — Kiara

Setting the mode

  • "resume" — pick up from logs/state, no re-briefing.
  • "autonomous, task-by-task, same mode as last sprint" — run the plan, don't check in per step.
  • "planning only — one question at a time, confidence % each turn, no code until go" — brainstorm; hold implementation.
  • "ground yourself on current truth first" — re-verify before proposing.

Authorizing an action

  • "go" (or "go —") — single green light. Silence ≠ yes.
  • "yes" — approve exactly the thing proposed, nothing extra.
  • "go + rider: stop only for genuine forks" — full autonomy, bounded.
  • "consider that a standing instruction going forward" — promote one-off to default.

Delegating

  • Terse target + trust: "put this project on the lab-os fork" — work out mechanics from precedent.
  • "walk me through it at that point" — operator/hands-on step is mine.

Being interviewed

  • Pick-one + rider: "option 2 — but don't block X on it."
  • "hold on —" — stop; the question is built on a wrong assumption.
  • "both" — the either/or is false.

Keeping it honest

  • "confirm with evidence before I retry" / "worktree, unmerged branch, or uncommitted?"
  • "tell me what correct looks like, so I can verify against expectations, not vibes"
  • Repeating the symptom flatly — an explanation isn't a fix.

Commands & rituals

  • "full sprint-close verification, evidence at every close"
  • "log this / add it to the deviation log"
  • The rhythm: plan file → task-by-task → close verification → log.
