# Getting Started

This exercise onboards you to agentic development — working with an AI that carries out tasks,
not just answers questions. You'll set up an agentic workspace — Claude Code (Anthropic's AI
coding assistant), persistent context files (`CLAUDE.md` layers), your own **fork of the lab-os
handbook as your development home** (its rules load into every session), and two plugins (add-on
capabilities for Claude Code). The setup itself is Claude-guided: you paste one prompt, Claude
interviews you and performs the steps — your first agentic collaboration.

`<DEV_ROOT>` is your development home — the local clone of your lab-os fork. Pick a path and use it
consistently — **Windows:** `C:\Users\<you>\Development\lab-os` · **macOS / Linux:**
`~/Development/lab-os`. Your own projects live in their own repos nested inside it.

> **Reading the commands:** blocks titled **Terminal** run in your OS terminal;
> blocks titled **Claude session** are typed into an active Claude Code session.

clone fallback.

## Step 1 — Install Claude Code

Follow [docs.claude.com/en/docs/claude-code/setup](https://docs.claude.com/en/docs/claude-code/setup),
confirm with `claude --version`, then run `claude` once and log in with your Claude account.

## Step 2 — Paste the bootstrap prompt

Start a session with `claude` and paste the entire block below. Claude interviews you first, then
performs the setup, confirming before every step that forks, clones, deletes, moves, or overwrites
files. The templates in
[lab-os/templates/](https://github.com/CAMELS-Research-Group/lab-os/tree/main/templates) stay the source of
truth — the prompt has Claude read them from your cloned fork.

```text title="Claude session"
I'm setting up my primary agentic development home as a fork of the lab-os
handbook. Walk me through setup interactively. Rules for the whole session:

- Interview me BEFORE doing anything: my name, career stage / background,
  what I'll be working on, my stack and primary shell, my working style, my
  OS (Windows / macOS / Linux), my time zone, where I want my dev home
  (<DEV_ROOT> — default ~/Development/lab-os, or
  C:\Users\<me>\Development\lab-os on Windows), and where my current project
  plan lives (a path or repo URL — I produced it in the Planning workshop).
- Confirm with me before any step that creates a fork, clones, deletes,
  moves, overwrites, or links files. If a target file already exists
  (especially ~/.claude/CLAUDE.md), show me a merge plan and get my OK —
  never silently overwrite.
- The lab-os templates are the source of truth. Read them from my cloned
  fork and follow their embedded instructions; do not improvise from memory.

Then do the following, in order, confirming each step with me:

1. Fork the handbook to my GitHub account and clone the fork into <DEV_ROOT>
   (the trailing path makes the clone land there directly), keeping an
   `upstream` remote pointed at the source for rule updates:
     gh repo fork CAMELS-Research-Group/lab-os --clone -- <DEV_ROOT>
   This sets origin -> my fork and upstream -> CAMELS-Research-Group/lab-os; confirm
   with `git -C <DEV_ROOT> remote -v`. If I have no GitHub account or decline
   to fork, fall back to:
     git clone https://github.com/CAMELS-Research-Group/lab-os.git <DEV_ROOT>
     git -C <DEV_ROOT> remote add upstream https://github.com/CAMELS-Research-Group/lab-os.git

2. Personalize <DEV_ROOT>/templates/global-CLAUDE.template.md into my global
   config at ~/.claude/CLAUDE.md (Windows: C:\Users\<me>\.claude\CLAUDE.md).
   Fill the About Me block from the interview; adapt the sections below About
   Me to my context; remove the template's instructional blockquote. If I
   already have a global CLAUDE.md, merge instead of overwriting — keep my
   About Me, fold in the adapted sections.

3. Seed <DEV_ROOT>/.claude/CLAUDE.md from
   <DEV_ROOT>/templates/dev-root-CLAUDE.template.md, adjusting paths to my
   machine and removing the instructional blockquote. (The lab rules already
   live natively at <DEV_ROOT>/.claude/rules — this is a fork, so there is NO
   junction to create; `git pull upstream main` is how I keep them current.)

4. Make this fork mine, not a copy of lab-os — confirm before deleting:
   - Reset <DEV_ROOT>/project_log.md (and project_log_archive.md if present)
     to a fresh, empty log from <DEV_ROOT>/templates/project_log.template.md.
   - Remove <DEV_ROOT>/docs/superpowers/specs/ (lab-os's own design history)
     and replace <DEV_ROOT>/README.md with a short one naming this as my dev
     home, keeping a one-line pointer to the upstream handbook.
   - Remove <DEV_ROOT>/site/ — the published handbook (workshop pages, guides)
     stays live upstream and I read it in the browser; only docs/-tier files
     like the sample plan are used locally, and those stay.
   - Stage this cleanup on a branch and note I can open it as my first PR.

5. Re-home my plan, and nest my project code if I brought any — confirm
   before moving:
   - Move my plan from the location I gave in the interview into
     <DEV_ROOT>/_plans/. It is tracked by the fork itself, alongside the rules
     and project log, so a session opened at my dev home sees all my plans.
   - If I brought an existing codebase, nest it as its own git repo at
     <DEV_ROOT>/<project>/ and add /<project>/ to <DEV_ROOT>/.gitignore, so the
     code keeps its own history and never tangles with `git pull upstream`.
     (No code yet? Skip this — create the project repo when I start building.)

6. Install the plugins: run `/plugin marketplace add WatsonWBlair/lab-claude-plugins`
   then `/plugin install pr-review-loop@lab-claude-plugins`. Also install
   `/plugin install superpowers@claude-plugins-official` (superpowers is on
   the official marketplace that ships with Claude Code — no marketplace add
   needed). (These are slash commands I run in Claude Code myself — walk me
   through them rather than running them for me.)

7. Run a verification pass and show me the results:
   - <DEV_ROOT>/.claude/rules lists the rule files (01-workflow.md,
     02-data-protection.md, ...)
   - git -C <DEV_ROOT> remote -v shows origin = my fork, upstream = the source
   - ~/.claude/CLAUDE.md exists with no remaining <...> placeholders
   - <DEV_ROOT>/.claude/CLAUDE.md exists; project_log.md is the fresh stub;
     site/ and docs/superpowers/specs/ are gone
   - my plan is at <DEV_ROOT>/_plans/, tracked by the fork; any codebase I
     brought is nested at <DEV_ROOT>/<project>/ as its own gitignored repo
   - the plugins are listed by /plugin
   Then tell me to start a fresh session at <DEV_ROOT> (my dev home — where I
   root my work; rules load natively here) and ask "what are the lab's
   commit-message rules?" — the answer should come from 01-workflow.md.

If anything fails, stop and explain before moving on. When everything
passes, summarize what was set up and where.
```

<details>
<summary>New to this? How your fork stays current with the lab rules</summary>

Your dev home is a *fork* of lab-os, so the rules at `<DEV_ROOT>/.claude/rules` are really yours —
no copy, no link to manage. To pull in rule updates from the upstream handbook later, run
`git pull upstream main` in `<DEV_ROOT>`. (The fork keeps an `upstream` remote pointing at
`CAMELS-Research-Group/lab-os` for exactly this.)

**Power user — many repos under one home?** If you'd rather keep several project repos side by side
under a neutral `<DEV_ROOT>` and share one rules source between them, clone lab-os as a
subdirectory and *link* its rules up to the dev root with a junction (Windows) or symlink
(macOS/Linux): `<DEV_ROOT>/.claude/rules` → `<DEV_ROOT>/lab-os/.claude/rules`. That's the lab's own
multi-repo setup; the fork-as-home flow above is the simpler default.

</details>

<details>
<summary>New to this? The three CLAUDE.md layers</summary>

`CLAUDE.md` files are instruction notes that Claude reads automatically at the start of every
session. The setup uses three, stacked from general to specific: your **global** file (who you are
and how you like to work — applies everywhere), the **dev-root** file (a map of your dev home —
applies when you work in `<DEV_ROOT>`), and a **per-repo** file (details of one project — applies
inside that project). Claude reads every layer that applies to where you opened the session, and
when two disagree, the most specific one wins. The bootstrap prompt sets up the first two (global +
your dev-home fork); each project you nest inside your home brings its own third.

</details>

## Verify your setup

```bash title="Terminal"
git -C <DEV_ROOT> remote -v
```

Expected: `origin` points at your fork; `upstream` points at `CAMELS-Research-Group/lab-os`. Your rules live
in `<DEV_ROOT>/.claude/rules` directly (no junction to check).

Then, in an active Claude Code session:

```text title="Claude session"
/plugin
```

Expected: lists `pr-review-loop@lab-claude-plugins` and `superpowers@claude-plugins-official`.

Start a fresh session at `<DEV_ROOT>` and ask:

```text title="Claude session"
what are the lab's commit-message rules?
```

Expected: answers from `01-workflow.md` (e.g. `feat:`, `fix:`, lowercase subject). Root your work
sessions at `<DEV_ROOT>` (your dev home), where the rules load natively; your nested project is a
subdirectory you work on from there.

Also confirm your global `~/.claude/CLAUDE.md` has no `<...>` placeholders left — About Me should
reflect you.

**If a check fails:** remotes wrong → re-run the prompt's fork/clone step (step 1). Rules question
goes unanswered → confirm the session was opened within `<DEV_ROOT>` (your fork root). Placeholders
still showing → finish personalizing the global file.
